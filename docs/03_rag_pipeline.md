# Task 3: RAG Pipeline — NOVA Product Knowledge Base

## What This Does

Customers ask questions about products — ingredients, sizing, compatibility, recommendations. Instead of relying on the LLM's training data (which may be wrong or outdated), we retrieve real product info from NOVA's catalog and feed it as context.

## How It Works

```
Customer: "What moisturizer is good for dry sensitive skin?"
    |
    v
Hybrid Search
    |---> Vector search (ChromaDB + MiniLM embeddings)
    |     Finds semantically similar products
    |
    |---> BM25 keyword search
    |     Finds exact term matches ("dry", "sensitive", "moisturizer")
    |
    v
Reciprocal Rank Fusion
    |  Merges both result sets, favoring products that appear in both
    v
Top 5 Products (with full details)
    |
    v
LLM generates answer grounded in actual product data
```

## Why Hybrid Search?

Neither approach alone is good enough:

| Search Type | Good At | Bad At |
|-------------|---------|--------|
| **Vector** (embeddings) | "Something for tired skin" → finds "revitalizing cream" | Misses exact terms like "hyaluronic acid" |
| **BM25** (keywords) | "hyaluronic acid" → exact match | Misses semantics: "moisturizing" ≠ "hydrating" |
| **Hybrid** | Gets both exact matches AND semantic matches | Slightly more complex to implement |

### Reciprocal Rank Fusion (RRF)
We merge results using RRF instead of raw score combination because vector and BM25 scores are on different scales. RRF uses rank position instead of raw score:

```
RRF_score(doc) = 1/(k + rank_vector) + 1/(k + rank_bm25)
```

Documents that rank high in both lists get the highest combined score.

## Components

### Embeddings
- Model: `all-MiniLM-L6-v2` (via ChromaDB's default)
- Runs locally — no API calls needed
- 384-dimensional vectors, fast and lightweight

### Vector Store: ChromaDB
- Embedded mode — no separate server, works in Colab
- Persistent storage so we don't re-index every run
- 200 products indexed with full text representations

### Document Chunking Strategy
Each product becomes one document containing all its attributes:
- Name, category, type, price, rating
- Ingredients (skincare), shades (makeup), sizes (apparel)
- Fit notes, hair types, skin types — whatever's relevant

We don't chunk products into smaller pieces because each product doc is already small (~200 tokens). Chunking would lose the relationship between attributes.
